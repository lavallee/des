# Operator mode

Use for data-heavy admin tools, queues, workbenches, monitoring, configuration,
and review surfaces used repeatedly by practiced professionals.

## Default contract

- Variance `low`; density `compact`; motion `functional`; type register
  `interface`; imagery `evidence-only`.
- Preserve task anatomy across routes. Stable placement is learned affordance,
  not generic design.
- Put identifiers, state, ownership, freshness, evidence, available verbs, and
  consequences close to the object they govern.
- Use tables for comparison, typed records for heterogeneous work, and an
  inspector or detail route for deciding evidence. A grid of generic cards is
  not an information model.
- Keyboard order, focus visibility, selection continuity, drafts, undo, and
  recovery are first-class states.
- Reserve human attention for judgment; do not display a machine backlog as a
  human obligation.

## Mode constraints

- Prefer flat, discrete surfaces. Do not use decorative gradients or shadows on
  task cards and controls; elevation is for actual overlays.
- Motion communicates state change, causality, or continuity. No scroll theater,
  looping decoration, or layout movement that shifts a target during work.
- Use one interface family plus mono where data benefits. A display face may
  mark product identity, but must not compete with operational hierarchy.
- Avoid decorative imagery. Screenshots, maps, charts, and thumbnails must
  provide evidence or orientation.
- Sentence case is the default. Metadata labels may use compact uppercase when
  that pattern remains readable.

Read `PATTERNS-ADMIN.md` and use the admin gates in `RUBRIC.md` for consequential
operator decisions.
